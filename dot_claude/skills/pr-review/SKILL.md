# pr-review — 5視点並列PRレビュー

GitHub PRを5つの専門家ペルソナが並列でレビューし、ハレーションチェックを経た統合レポートを生成する。

## ステップ 1: PR特定

`$ARGUMENTS` を解析する:

- 数値のみ（例: `123`）→ そのままPR番号として使用
- GitHub URL（`/pull/123` を含む）→ URLから番号を抽出
- 引数なし → `gh pr view --json number` で現在のブランチのPRを取得

PR番号が特定できない場合はユーザーに確認して停止する。

## ステップ 2: PRデータ一括収集

以下を並列で実行してPRの全情報を収集する:

```bash
# 基本情報・説明・ラベル
gh pr view <PR番号> --json number,title,body,author,baseRefName,headRefName,labels,assignees,reviewRequests,state,createdAt,updatedAt

# コミット履歴
gh pr view <PR番号> --json commits --jq '.commits[] | "\(.messageHeadline)\n  \(.messageBody // "")"'

# 変更ファイル一覧（追加/変更/削除の状態付き）
gh pr diff <PR番号> --name-only

# 差分全体
gh pr diff <PR番号>

# 既存レビューコメント（本文レベル）
gh pr view <PR番号> --json reviews --jq '.reviews[] | "[\(.author.login)] \(.state): \(.body)"'

# インラインレビューコメント（ファイル:行レベル）
gh pr view <PR番号> --json reviewThreads --jq '.reviewThreads[] | .comments[] | "[\(.author.login)] \(.path):\(.line // .originalLine // "?"): \(.body)"'

# 既存コメント（PRスレッド）
gh pr view <PR番号> --json comments --jq '.comments[] | "[\(.author.login)]: \(.body)"'
```

> **大規模PR対応**: 差分が10,000行を超える場合は最初の8,000行を使用し、残りは「以降省略（大規模差分）」と注記する。

収集したデータを以下の変数として組み立てる（後続ステップで使用）:

```
PR_DATA = """
## PR基本情報
タイトル: <title>
番号: #<number>
著者: <author>
状態: <state>
ベースブランチ: <baseRefName>
ヘッドブランチ: <headRefName>
ラベル: <labels>
作成日時: <createdAt>

## PR説明
<body>

## コミット履歴
<commits>

## 変更ファイル一覧
<changed files>

## 差分
<diff>

## 既存レビューコメント（本文レベル）
<reviews>

## 既存インラインコメント（ファイル:行レベル）
<reviewThreads inline comments>

## 既存コメント（PRスレッド）
<comments>
"""
```

## ステップ 4: 5サブエージェント完全並列起動

**以下の5つのAgentツール呼び出しを同一ターンで同時に発行する。** 順番に実行してはならない。

各サブエージェントに渡すプロンプトは次のテンプレートで構築する:

---
### サブエージェント共通プロンプトテンプレート

```
あなたは [ペルソナファイルに記載のタイトル] です。
以下の指示に従い、提供されたGitHub PRを厳格にレビューしてください。

=== 【必須】既存コメントの確認 ===
PRデータに「既存レビューコメント（本文レベル）」「既存インラインコメント（ファイル:行レベル）」
「既存コメント（PRスレッド）」が含まれています。
レビュー前にこれらを必ず確認し、以下のルールに従うこと:

1. 既に指摘されている内容は重複して指摘しない
2. 既存指摘を補完・深掘りする場合は冒頭に「既存コメント（@著者名）への追記:」と明示する
3. 既存コメントで解決済み（resolved）のスレッドの内容は指摘しない

=== レビュアー指示 ===
[ペルソナファイル (pe.md / pm.md / staff.md / req.md / security.md) の全内容]

=== PRデータ ===
[PR_DATA の全内容]

=== 出力要件 ===

## メタ情報タグ（必ず使用すること）
| タグ | 期待対応 | 説明 |
|------|----------|------|
| [ask] | 回答必須 | 確認が必要な事項 |
| [must] | 修正必須 | これがなければApprove不可 |
| [imo] | 修正任意 | 対応なくてもApprove可 |
| [nits] | 修正任意 | 細かい指摘 |
| [next] | 修正不要 | 今後の改善点 |
| [good] | — | 良い点（積極的に言及する） |
| [suggestion] | — | 代替案の提案 |

## 指摘フォーマット
各指摘は必ず以下を含めること:
- メタ情報タグ（上記から1つ選択）
- **ファイル名と行番号**（例: `src/api/user.ts:42`）
- 問題の説明（「何が」「なぜ問題か」）
- 改善案（具体的なコードスニペットまたは手順）
- **【必須】権威ある参照** — [must]/[imo]/[ask]/[nits] タグには以下のいずれかを必ず1件以上記載すること:
  - 公式ドキュメントURL（言語・フレームワーク・ライブラリの公式サイト）
  - RFC・W3C・IEEE・ISO 等の標準規格（番号 + URL）
  - OWASP・NIST・CWE・CVE 等のセキュリティ標準
  - Google/Airbnb/Microsoft 等の業界標準スタイルガイド
  権威ある参照を提示できない場合は [suggestion] タグに格下げし
  「個人的見解のため参照なし」と明記すること。
  [good]/[next]/[suggestion] は参照任意。

## 出力フォーマット
以下のMarkdown構造で返すこと（見出しレベルを変えないこと）:

### [ペルソナ名] レビュー

#### 総評
（このPRに対する全体的な評価を2〜4文で記述）

#### 指摘事項
（各指摘を箇条書きで。深刻なものから順に記載）

#### 良い点
（[good] タグで評価できる点を具体的に）
```
---

各ペルソナファイルの内容をそのままプロンプトに埋め込み、PR_DATAもフルで渡す。

> **重要**: 5つのAgentツール呼び出しは必ず同時に発行すること。先行するエージェントの完了を待ってから次を起動してはならない。

## ステップ 5: 全エージェント完了待機・結果収集

5つのエージェントがすべて完了するまで待機し、それぞれの出力を収集する。

各エージェントの出力を以下の変数に格納:
- `RESULT_PE` — Principal Engineerレビュー結果
- `RESULT_PM` — PMレビュー結果
- `RESULT_STAFF` — Staff Engineerレビュー結果
- `RESULT_REQ` — 要求工学レビュー結果
- `RESULT_SECURITY` — セキュリティレビュー結果

## ステップ 6: 妥当性検証（一次フィルタリング）

5つの結果を俯瞰し、以下の観点で各指摘を粗くフィルタリングする（精密なチェックは次ステップ）:

**除外すべき指摘（ノイズ）:**
- 汎用的すぎて具体性がない（例: 「セキュリティを考慮してください」のみ）
- すでに他の視点で同一内容が指摘されており、重複と判断できるもの
- PRの変更範囲を大きく超えた将来の推測に過ぎないもの

**格上げ検討 ([imo] → [must]):**
- 複数の視点で共通して重大と評価された指摘

**格下げ検討 ([must] → [imo]):**
- PRの説明で既に対処済みと明記されているもの

一次フィルタリング結果を内部メモとして保持し、次ステップへ渡す。

## ステップ 7: ハレーションチェックサブエージェント

Agentツールを使って独立したファクトチェッカーエージェントを1つ起動する。
以下のプロンプトで実行する:

---
```
あなたは独立したファクトチェッカーです。感情を持たない検証専門エージェントとして、
提供された5つのPRレビュー結果を、実際のPR差分データと厳密に照合してください。

=== 検証対象レビュー ===
[RESULT_PE / RESULT_PM / RESULT_STAFF / RESULT_REQ / RESULT_SECURITY の全内容]

=== 照合用PRデータ ===
[PR_DATA の全内容（差分・既存コメント・インラインコメントを含む）]

=== 検証項目 ===

【1】コード存在確認
- 指摘されているファイル名が「変更ファイル一覧」に存在するか確認する
- 指摘されている行番号が差分の当該ファイルの変更範囲（+/-行）内にあるか確認する
- 存在しないファイル・範囲外の行を指摘しているものは「コード不在」として報告する

【2】参照URLの実在確認（WebFetch を使用）
- [must] タグの指摘に記載されたURLを実際にフェッチして HTTP 200 を確認する
- 404・接続不可・タイムアウトの場合は「参照不在」として報告する
- [must] 指摘に参照URLが記載されていない場合は「参照欠落」として報告する
- URLが多数ある場合は [must] のみ対象とし、[imo] 以下はサンプル確認でよい

【3】既存コメントとの重複確認
- 「既存レビューコメント」「既存インラインコメント」「既存コメント」に
  実質的に同一の内容が既に指摘されている場合は「既存重複」として報告する
- 同一かどうかは「指摘の本質的な意図」で判断し、表現の違いは重複とみなさない

【4】主張の論理確認
- 指摘内容がdiffのコードに実際に当てはまるか確認する
- 「〜のリスクがある」という指摘について、そのリスクを引き起こすコードが
  実際にdiffに存在するか確認する
- diffに存在しないリスクを断言している指摘は「論理不整合」として報告する

=== 出力フォーマット（このフォーマットを厳守すること） ===

### ハレーションチェック結果

#### 削除推奨（事実と異なる指摘）
| 視点 | 指摘の概要 | 理由 |
|------|-----------|------|
| PE   | ...       | コード不在: `src/api/user.ts:42` はdiffに存在しない |

削除推奨がない場合は「削除推奨なし」と記載する。

#### 格下げ推奨（参照欠落または根拠不十分）
| 視点 | 元のタグ | 推奨タグ | 理由 |
|------|---------|---------|------|
| SEC  | [must]  | [imo]   | 参照欠落: 権威ある参照が記載されていない |

格下げ推奨がない場合は「格下げ推奨なし」と記載する。

#### 既存コメント重複
| 視点 | 指摘の概要 | 重複している既存コメント（著者・内容） |
|------|-----------|--------------------------------------|

重複なしの場合は「重複なし」と記載する。

#### 検証サマリー
- 検証した指摘の総件数: N件
- 問題なし（そのまま採用可）: N件
- 削除推奨: N件
- 格下げ推奨: N件
- 既存重複: N件
```
---

ハレーションチェック結果を `RESULT_HALLCHECK` として収集する。

## ステップ 8: 統合レポート生成・出力

`RESULT_HALLCHECK` の結果を適用してから、以下の構造でMarkdownレポートを生成し端末に出力する:

**ハレーションチェック結果の適用ルール:**
- 削除推奨 → レポートから除外
- 格下げ推奨 → 指定タグに変更して反映
- 既存重複 → 対象指摘の末尾に「※ 既存コメント（@著者名）にて既出」を付記

---

```markdown
# PR Review Report

## #<番号> <タイトル>
**著者**: <author> | **ベース**: <baseRefName> ← <headRefName> | **状態**: <state>

---

## Executive Summary

> （5視点全体を踏まえた総評。このPRのマージ可否判断に直接役立つ3〜5文。
> [must]が存在するかどうかを明示する）

---

## [must] 修正必須項目（全視点集約）

> ※ このセクションの項目がすべて解消されるまでApproveを推奨しない
```
